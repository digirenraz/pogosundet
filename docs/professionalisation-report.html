P0

1 · Run the type-check (and build) in CI

Impact: Very highEffort: 10 minCost: FreeCI

Why it matters: Your CLAUDE.md says "npm run build is the canonical type gate" — but CI runs only lint, vitest, and Playwright (which boots next dev, not a production build). Nothing in CI ever type-checks or builds. In a workflow where a human seldom reads the diff, a TypeScript error or a broken production build can be merged and only discovered when prod breaks. This is the cheapest high-value fix on the list.

Technical detail

Add a typecheck script and two CI steps. tsc --noEmit is the fast gate; next build additionally catches build-time failures (RSC boundaries, route config) that tsc misses.

// package.json
"scripts": {
  "typecheck": "tsc --noEmit",
  ...
}
# .github/workflows/ci.yml — add before the e2e step
      - name: Typecheck
        run: npm run typecheck

      - name: Build
        run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.NEXT_PUBLIC_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.NEXT_PUBLIC_SUPABASE_ANON_KEY }}

Optional but recommended: have Playwright test against the production build instead of next dev (change webServer.command to npm run build && npm start in CI) so e2e exercises the same artefact users get.

2 · Stand up a free second Supabase project as staging/preview

Impact: Very highEffort: 1–2 hrsCost: Free (Supabase free tier allows 2 projects)Environments

Why it matters: This is the risk your own "Open questions" flags twice. Today, profiles, raids, and chat created on a Vercel preview deploy land in the production database, and any data-writing test pollutes real data. It also forces the brittle "apply migrations to the live DB before merge" dance, and it's the reason logged-in e2e tests can't run in CI. A separate preview DB dissolves three problems at once — and it's free.

Technical detail
  • Create a second Supabase project (EU/Ireland, same as prod) — call it pogosundet-preview.
  • Apply all 21 migrations to it once (paste in order) so its schema matches prod.
  • In Vercel, set Preview-scoped overrides for NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY pointing at the preview project (Production env vars stay on prod). Vercel supports per-environment values natively.
  • Point CI's Supabase secrets at the preview project too — then CI e2e no longer touches prod (prerequisite for #9).
  • Now "apply-before-merge" becomes "apply to preview, verify on the preview deploy, then apply to prod at merge" — a real staging gate instead of editing the live DB.

Caveat to keep: Edge Functions / push webhooks are still wired to the prod project, so push delivery still needs a prod check — but every data flow becomes safely testable.

4 · Make CI a required status check on main

Impact: HighEffort: 5 minCost: FreeGitHub policy

Why it matters: Your branch ruleset requires a PR but 0 approvals, and Claude self-merges. That's a reasonable choice for a solo vibe-coded project — but only if CI is a required check. If CI is merely advisory, a self-merge can land a red build. Combined with #1 (adding the type gate to CI), making CI required is what actually closes the "broken code can't reach main" loop.

Technical detail

In the GitHub ruleset Protect main — require PR (id 17923605), enable "Require status checks to pass" and add the test job from ci.yml as a required check. Keep approvals at 0 so self-merge still works. The Claude review stays advisory (correct — it's a judgement aid, not a pass/fail gate). Verify by opening a PR with a deliberate failing test and confirming the merge button is blocked.

Quick way to confirm current state: GitHub → Settings → Rules → Rulesets → Protect main → check whether "Require status checks" is listed. If it isn't, that's the gap.

5 · Enable secret scanning + push protection

Impact: Medium-highEffort: 5 minCost: Free (public repo)Security

Why it matters: The repo was made public on 2026-06-20 to unlock free branch protection. Public repos get GitHub's secret scanning and push protection for free — push protection actively blocks a commit that contains a key before it ever reaches the public history. Given the whole security posture depends on "no secrets in code," this turns that rule from a manual promise into an enforced guarantee. Especially valuable when an AI is generating commits.

Technical detail

GitHub → Settings → Code security → enable Secret scanning and Push protection. Both are free for public repositories. Push protection rejects pushes containing recognised credential patterns (Supabase service keys, GitHub PATs, Anthropic keys, etc.) with an override-with-justification flow. No code change.

6 · CI guard for the manual-migration workflow

Impact: HighEffort: 30–60 minCost: FreeMigrations

Why it matters: Your most dangerous class of bug is "a PR ships a query against a column the shared prod DB doesn't have yet" — it breaks the app for every user the instant it deploys. Right now the only defence is a human remembering the "apply-before-merge" rule. A cheap CI guard turns that into an explicit, un-skippable prompt.

Technical detail

Two layers, both free:

  • CI label/comment guard: a small workflow step that, when a PR's diff touches supabase/migrations/**, fails (or posts a sticky comment) unless the PR body contains a checked "Migration applied to prod ✅" box. Forces the author to acknowledge it.
  • Better, once #2 (staging DB) exists: manage migrations with the Supabase CLI (supabase migration new, supabase db push) against the preview project, so the preview deploy proves the migration applies cleanly before prod. You already use the CLI for Edge Functions; this extends it to schema.

This is also where the friend-code RLS residual (#18) would naturally get fixed, since you'd have a safe place to test a view/RPC refactor.

8 · Add Dependabot

Impact: MediumEffort: 10 minCost: FreeGitHub policy

Why it matters: You're on the bleeding edge (Next 16, React 19) with no automated dependency or vulnerability tracking. Dependabot opens PRs for security patches and version bumps — and because those PRs run through your CI + AI reviewer, an AI can triage and merge the safe ones with the same workflow you already use. Keeps a vibe-coded app from silently rotting.

Technical detail
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "npm"
    directory: "/"
    schedule: { interval: "weekly" }
    open-pull-requests-limit: 5
    groups:
      minor-and-patch:
        update-types: ["minor", "patch"]
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule: { interval: "weekly" }

Grouping minor/patch bumps into one PR keeps the noise down. Also enable Dependabot security alerts under Settings → Code security (free, public repo).

18 · Close the friend-code column-RLS residual

Impact: Medium (privacy)Effort: 1–2 hrsCost: FreeSecurity

Why it matters: Documented in your decisions log: the "hide friend code" feature redacts at the page layer, but a technical user can still read any friend_code directly via the Supabase REST API with the public anon key, because RLS is row-level and the SELECT policy returns all columns. The in-app feature is honoured; the raw API isn't. Worth closing before launch since it's the one known way the privacy promise leaks.

Technical detail

Proper fix is column-level masking: expose profiles through a view or RPC that omits friend_code for rows where hide_friend_code is true (and the viewer isn't the owner), and lock direct table SELECT down to the owner. This is exactly the kind of schema change that's risky on a shared prod DB — so it's best done after #2 (staging) and #6 (CLI migrations) give you a safe place to test it. Flag as a follow-up to security-review Finding 3.
