How the project is built & how changes ship

A plain-language tour of the tech stack, the test safety net, and the pull-request flow — written as an intro for a fellow (semi-technical) PM. Internal reference; not part of the app.

Mobile-first PWA Danish community · Frederikssund EU-hosted (GDPR) AI-reviewed pull requests

PoGoSundet is a mobile web app that helps the local Pokémon GO community find each other, share Trainer Codes, and coordinate raids. It installs to a phone's home screen like a native app and sends push notifications. One codebase, built and verified by Claude Code with the PM steering.

1The tech stack

Each layer was chosen once and is locked unless there's a concrete reason to change. No custom servers — managed services do the heavy lifting.

Frontend
Next.js 16 + React 19

The app framework. Mobile-first, renders pages on the server for speed. Tailwind CSS for styling.

Backend & Database
Supabase

Postgres database, login, file storage, live updates, and small server functions — all in the EU/Ireland region for GDPR.

Login
Supabase Auth

Google sign-in and email/password. Security rules live in the database itself.

Hosting
Vercel

Deploys automatically. Servers run in Dublin (dub1) next to the database to keep things fast.

Language / i18n
next-intl

Danish-first. Every piece of text lives in a translation file — no words are hard-coded in screens.

App & Push
PWA + Web Push

A hand-written service worker makes it installable and delivers notifications (new raids, DMs, raid chat).

Analytics & Errors
Amplitude · Sentry

Opt-in, anonymous product analytics and error logging. Both stay off until a consent banner is accepted / a key is set.

Testing
Vitest · Playwright

Vitest checks logic; Playwright drives a real browser through user flows. More on this below.

3What "CI" actually runs

CI ("continuous integration") is the automated test runner. It fires on every PR and every merge, on a clean machine, in this order. If any stage fails, the PR is blocked from merging.

Lint

Catches code-style and obvious mistakes automatically.

→
Unit tests · Vitest

240+ fast checks on logic: validation, helpers, calculations.

→
Browser tests · Playwright

Drives a real Chrome browser through user journeys end-to-end.

Runs on Node 24 / Ubuntu. The two AI checks (this CI + the PR reviewer) are independent: CI proves the code works; the reviewer judges whether it's well-written and follows our project rules. Tests, not opinions, versus opinions, not tests.

4Test coverage — what's guarded, what's verified by hand

Two layers of automated tests, plus a deliberate manual step for things a test machine genuinely can't see.

Unit tests (Vitest)

Tiny, instant checks on pure logic. Written test-first (TDD).

  • Validation — profile fields, raid forms, message length limits
  • Helpers — gym distance/sorting, friend-code redaction, participant lists
  • Data shaping — chat message mapping, unread counts
  • 240+ tests, co-located next to the code they cover

Flow tests (Playwright)

A real browser clicks through full user journeys. ~18 flow files.

  • Logged-out flows run in CI: home, login CTAs, privacy, loading screen, consent banner
  • Logged-in flows — chat, DMs, raids, profile edit, hide friend code
  • Logged-in specs skip in CI — they need test credentials CI doesn't have (they'd write into the shared database), so they run on a dev machine
Verified on production, not the test machine: live features — push notifications, real-time chat delivery, and actions only a raid's host can do — behave differently (or not at all) on a local test box. These are confirmed by hand on the live app, often with a second account. It's a documented rule, not an oversight.

5Project guardrails the AI reviewer enforces

The PR reviewer is primed with this project's specific rules (kept in CLAUDE.md), so it flags more than generic bugs:

EU region on new server code
Every new server route must run in Dublin (dub1) — far from the EU database is slow.
No hard-coded text
All UI words go through the Danish translation file, never typed into a screen.
GDPR on new personal data
A new data field or third-party service must update the Privacy Policy.
Database migrations before merge
Schema changes are applied to the shared database before the PR merges, or it breaks.
App-version bump on app-shell changes
Touching the service worker bumps a cache version so installed phones update.
User-facing changelog
Features and visible fixes add a short Danish "Nyheder" entry for users.

6Environments & a known gap

How code reaches users

Two automatic deploys per change.

  • Preview — every PR gets its own throwaway URL to click around on
  • Production — merging to main deploys the live app
  • Database changes are applied manually in Supabase's SQL editor (no auto-runner, by design)

The one gap to know about

Worth keeping in mind before launch.

  • No separate staging database — preview and production share one Supabase project, so test data created on a preview is visible in production
  • This is why logged-in tests don't run in CI, and why risky data changes are double-checked by hand
