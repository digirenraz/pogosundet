PoGoSundet

Project overview — what's built and how it works

A self-contained reference for the current state of the app: features, tech stack, architecture invariants, and how changes ship safely. Internal document; not part of the app.

Mobile-first PWA Danish community · Frederikssund EU-hosted · GDPR AI-assisted development Last updated 2026-06-28

PoGoSundet is a mobile web app for the local Pokémon GO community in Frederikssund. Players create profiles, find each other, share Trainer Codes, and coordinate raids — faster than posting a screenshot to Messenger. It installs to the home screen like a native app and sends push notifications. One codebase, steered by the PM, built by Claude Code.

1What the app does

Everything below is live on production today.

👤
Player profiles & directory

Every member creates a profile with their trainer name, team, level, and Trainer Code. The directory lists all members with search and shows who's online. A QR code is generated per profile so codes can be scanned directly in Pokémon GO.

⚔️
Raid coordination

Post a raid with boss, gym, and time. Others join, indicate how many extra players they're bringing, and react (shiny / hundo). The host marks the raid done — sign-ups lock, reactions stay open. Copy all attendee names to clipboard for the lobby invite.

💬
Community chat

Two fixed channels: #generelt (general) and #app-feedback. Messages support emoji reactions and threaded replies. Live via Supabase Realtime — new messages appear without refreshing.

✉️
Direct messages

1:1 DMs between any two members. Same reaction and reply stack as channel chat. Entry points from the online strip, members sheet, and player directory. Push notification delivered to the recipient.

🔔
Push notifications

Web push via a hand-written service worker. Triggers: new raid posted, new raid message, someone joins a raid you're attending, new DM received. iOS requires Safari Add-to-Home-Screen. No notifications without install.

🗺️
Gym database

152 gyms in Frederikssund seeded from collect.dk (with permission). The raid form autocompletes gym names and suggests nearby and recent gyms first. "Vis på kort" opens the exact coordinates in the device's maps app.

🖥️
Desktop layout

At ≥1024 px, a persistent sidebar replaces the bottom nav. The friend-scan session shows each player's QR code full-size, one at a time, with keyboard navigation — designed to be open on a desktop while scanning codes on a phone. Progress persists between sessions.

📲
PWA install

Add to home screen on Android (Chrome) and iOS (Safari). Works offline for cached routes. The app icon is a branded medallion (not the Poké Ball, for IP reasons). Guided install flow at /onboarding.

🆘
In-app bug reports

A "Rapportér en fejl" item in the hamburger menu opens a form that creates a real GitHub issue in the repo, pre-filled with browser/OS info. Verified end-to-end on production.

📋
In-app changelog & help

A "Nyheder" log (user-facing changelog) shows the latest feature updates in plain Danish. A "Hjælp" section has 16 help articles covering every feature. Both are lazy-loaded sheets — not in the initial bundle.

🛡️
GDPR-compliant accounts

Consent checkbox at registration (not pre-ticked). Full account deletion with data cascade. Friend codes can be hidden. Analytics opt-in only. All data in the EU.

🔍
Friend-code privacy

Players can hide their Trainer Code from the directory. The column-level RLS fix (migration 022) means the code can't be read via the raw Supabase REST API either — not just the UI.

2Tech stack

Each choice is locked — do not switch without a concrete blocker and explicit agreement.

Frontend
Next.js 16 · React 19

App Router, server components, Tailwind CSS. Single codebase for mobile + desktop.

Backend & Database
Supabase

Postgres, Auth, Storage, Realtime, and Edge Functions — all in the EU/Ireland region for GDPR. 23 migrations applied.

Hosting
Vercel

Auto-deploys on every merge to main. All server functions run in Dublin (dub1), next to the database.

Auth
Supabase Auth

Google OAuth + email/password. Row-level security enforces access rules inside the database itself.

i18n
next-intl

Danish-first. Every UI string lives in messages/da.json. No text hard-coded in components.

PWA & Push
Hand-written SW + Web Push

No framework wrapper. public/sw.js (currently v17). Push delivered via four Supabase Edge Functions triggered by DB webhooks.

Error monitoring
Sentry

EU project, live on production. sendDefaultPii: false. Client errors captured and reported to the EU ingest endpoint.

Product analytics
Amplitude

Opt-in only — loads after the consent banner is accepted. EU server zone, fully anonymous (no user_id, no IP). Inactive until the PM sets the API key in Vercel.

Testing
Vitest · Playwright

Vitest for unit/logic tests; Playwright drives a real Chrome browser through end-to-end user flows. Both run in CI on every PR.

3Architecture — the invariants that must never break

These aren't style preferences — they're load-bearing rules. The AI code reviewer is primed to flag any violation.

🔌
Three Supabase clients — never mixed

client.ts for browser components, server.ts for server components and route handlers, admin.ts (service role) for privileged ops only. Using the wrong one either leaks credentials or silently fails.

🌍
All server functions run in Dublin

Every route handler and server page exports preferredRegion = "dub1". Supabase runs in AWS eu-west-1 (Ireland). The default US East region adds ~80 ms per query — on every page load.

🗄️
Migrations applied before the code merges

There is no migration runner. SQL goes into the Supabase SQL editor manually. If a PR references new columns that don't exist yet, every user sees an error the moment it deploys. A CI guard enforces the confirmation.

🔄
Middleware chain: session → i18n

The Next.js middleware file (proxy.ts) runs Supabase session refresh first, then next-intl locale routing. Auth cookies are manually copied onto the intl response to prevent loss between the two middlewares.

🔢
Service worker versioning

Every change to public/sw.js behaviour bumps the SHELL_CACHE / RUNTIME_CACHE version constant (currently v17). This evicts the old cache on installed PWAs. Forgetting it means users keep the old push handler.

🔐
Push Edge Functions verify a shared secret

All four notify-* Edge Functions check an x-webhook-secret header against WEBHOOK_SECRET in Deno env. Fail-open if unset (safe to deploy), enforced on production. New notify functions must replicate this gate.

🌐
Danish URLs have no locale prefix

localePrefix: 'as-needed' — Danish is the default, so URLs are /players, not /da/players. Pages live under src/app/[locale]/ in the file system, but the /da/ segment is stripped at the URL level.

💾
Player directory caching uses the admin client

getAllProfiles() uses unstable_cache (60 s TTL, tag profiles). The regular server client calls cookies(), which is unavailable outside a request context. Account deletion busts the cache with revalidateTag('profiles').

5How a change ships — the pull-request flow

Direct pushes to main are blocked by a branch ruleset. Every change — even a one-line typo fix — travels this path.

STEP 1
Branch

Work happens on an isolated copy. Naming convention: slice/N-name or chore/name.

Claude Code ›
STEP 2
Open a PR

Proposed change published with a plain-language summary and the invariant checklist (migration applied? changelog? SW bump? dub1?).

Claude Code ›
STEP 3
Automatic checks

CI pipeline runs. A fresh AI reviewer (separate Claude instance, primed with project rules) reads the diff — in parallel.

CI + AI review ›
STEP 4
Address findings

Valid issues get fixed; false positives are pushed back on. Pushes re-trigger CI automatically.

Claude Code ›
STEP 5
Squash-merge & deploy

Once CI is green, squash-merge to main. Vercel deploys automatically. Branch deleted.

Claude Code
The migration rule: if a PR introduces a database query against new columns or tables, those must be applied to the production Supabase SQL editor before the PR merges — not after. Merging first errors every user the moment Vercel deploys. A CI guard requires the author to check a box in the PR description confirming this was done.

7GDPR at a glance

Denmark is in the EU. These requirements are non-negotiable and already in place.

🇪🇺
EU-only data

Supabase runs in AWS eu-west-1 (Ireland). Vercel servers in Dublin. Analytics and error monitoring in EU regions.

✅
Explicit consent

Registration requires an un-pre-ticked consent checkbox. Analytics only activate after the in-app consent banner is accepted.

🗑️
Account deletion

Full cascade — deleting the auth user removes all profile data, messages, reactions, and scan records. Implemented at the API layer, not just UI.

🔒
Anonymous analytics

Amplitude: no user_id, no IP address, no PII. EU server zone. Opt-in gated behind a consent banner.

📄
Privacy Policy

In-app at /privacy, in Danish. Updated whenever a new data field or third-party service is introduced.

👁️
Friend-code privacy

Players can hide their Trainer Code. Enforced at the database column level (RLS REVOKE) — not just the UI. The raw REST API also withholds it.

8Deliberately not built (yet)

Keeping scope tight is intentional — these are Phase 2 items that would need significant design work before building.

Out of scope for the current phase: remote raid lobby code sharing, recurring raids or raids scheduled more than a few hours ahead, raid history or past-raid browsing, search or filter options on the raid list, host/organiser roles, admin or moderation tools, trade request coordination, and a language switcher (Danish-only for now).

The rule: if a feature would require significant refactoring to support Phase 2, flag it and ask first. Do not pre-build for hypothetical future requirements.

9How the team works

Primary implementation — builds, tests, reviews, merges

  • Writes code, tests, migrations, and documentation
  • Opens PRs with plain-language summaries
  • Watches CI, debugs failures, fixes issues
  • Merges PRs once CI is green (delegated by PM 2026-06-20)
  • Keeps CLAUDE.md and decisions log up to date each session
  • Updates the in-app changelog for every user-facing change
The source of truth for everything in this project is CLAUDE.md — it contains the architecture rules, open questions, decisions log, and next-up backlog. It is read automatically at the start of every Claude Code session. The decisions log at the bottom of that file is the running record of what was decided and why.
